# Shipyard Data Model — Single Source of Truth

## Data Location

**All Shipyard data lives outside the project** in the plugin data directory. Always resolve the path by running `shipyard-data` in Bash:

```bash
SHIPYARD=$(shipyard-data)
# Then use: $SHIPYARD/spec/features/, $SHIPYARD/config.md, etc.
```

**NEVER:**
- Hardcode `.shipyard/` paths (legacy — data moved to plugin data in v0.5.0)
- Guess absolute paths or look in other worktrees/directories
- Use paths from other branches or worktrees
- Use `status` as a shell variable name — it's read-only in fish shell. Use `fstatus`, `task_status`, or `feat_status` instead

**ALWAYS** use `$(shipyard-data)` to resolve the path dynamically.

## Source of Truth Hierarchy

Every piece of data has exactly ONE authoritative location. All other files are **generated views** rebuilt from source files. Never manually sync data between files — regenerate the view instead.

## Source of Truth Hierarchy

### Feature Files (`$(shipyard-data)/spec/features/FNNN-*.md`)
Authoritative for:
- `title`, `status`, `epic`, `story_points`, `complexity`, `feasibility`
- `rice_reach`, `rice_impact`, `rice_confidence`, `rice_effort`, `rice_score`
- `token_estimate`, `dependencies`, `references`, `children`
- User story, acceptance scenarios (Given/When/Then), decision log
- `tasks:` array in frontmatter — list of task IDs belonging to this feature

### Task Files (`$(shipyard-data)/spec/tasks/TNNN-*.md`)
Authoritative for:
- `title`, `status`, `effort`, `dependencies`, `feature` (parent ID)
- What to build, acceptance criteria, technical notes

### Epic Files (`$(shipyard-data)/spec/epics/ENNN-*.md`)
Authoritative for:
- `title`, `description`, `status`
- **NOT** authoritative for which features belong to it — that's derived from feature `epic:` fields

### Idea Files (`$(shipyard-data)/spec/ideas/IDEA-NNN-*.md`)
Authoritative for:
- `title`, `status`, `source`, notes

## Generated Views (rebuilt from source, never manually edited for data)

### BACKLOG.md (`$(shipyard-data)/backlog/BACKLOG.md`)
- **Contains**: Ordered list of feature IDs + manual override notes
- **Generated columns** (read from feature frontmatter on display): title, RICE, points, status, complexity
- **Only manually editable field**: rank order and override reasoning
- **Regenerated by**: Any skill that modifies feature status, estimates, or backlog membership

Structure:
```markdown
---
last_groomed: YYYY-MM-DD
---
# Backlog
| Rank | ID |
|------|----|
| 1 | F009 |
| 2 | F011 |
| 3 | F004 |

## Overrides
- F011 ranked above F004 despite lower RICE: [reasoning]
```

When displaying the backlog, read each feature file to get title, RICE, points, status, complexity. Never store these in BACKLOG.md.

### SPRINT.md (`$(shipyard-data)/sprints/current/SPRINT.md`)
- **Contains**: Sprint goal, capacity, wave structure (task IDs grouped by wave), execution mode
- **Generated data** (read from task/feature files on display): task titles, effort, status, feature titles
- **Sprint-owned data**: goal, capacity, wave groupings, critical path, swap log, dates
- **Regenerated by**: ship-sprint (plan), ship-execute (after task completion)

Structure:
```markdown
---
id: sprint-NNN
status: active
goal: ""
capacity: 20
branch: ""              # user's working branch — recorded by ship-execute on start
created: YYYY-MM-DD
started_at: null        # ISO 8601 — written by ship-execute on first wave start
completed_at: null      # ISO 8601 — written by ship-execute on sprint completion
total_paused_minutes: 0 # accumulated across all pause/resume cycles
---
# Sprint NNN

## Goal

## Waves
### Wave 1
- T001
- T002

### Wave 2
- T003
- T004

## Critical Path
T001 → T003 → T007

## Swap Log
| Date | Added | Removed | Reason |
```

When displaying the sprint, read each task file to get title, effort, status, feature parent. Never store these in SPRINT.md.

### PROGRESS.md (`$(shipyard-data)/sprints/current/PROGRESS.md`)
- **Contains**: Current wave number, blockers, session notes
- **Generated data** (read from task files on display): completion counts, task statuses
- **Progress-owned data**: current_wave, blockers, session log entries

Structure:
```markdown
---
current_wave: 1
---
# Sprint Progress

## Blockers
| Task | Reason | Since | Escalation |
|------|--------|-------|------------|

## Deviations
| Task | Type | What Changed | Why |
|------|------|--------------|-----|

## Patch Tasks (added during sprint)
| Task | Patches | Reason |
|------|---------|--------|

## Session Log
- [date] [summary of work done]
```

Task completion status lives in task file frontmatter (`status: done`). Count done tasks by reading task files, not by maintaining a counter.

## Status Workflow State Machine

One canonical set of valid transitions. All skills reference this — never hard-code transitions locally.

### Feature Status
```
proposed → approved → in-progress → done → deployed → released
proposed → deferred (park it)
proposed → rejected (kill it)
approved → deferred
in-progress → approved (pulled from sprint, back to backlog)
deferred → proposed (revisit)
```

### Task Status
```
approved → in-progress → done
approved → blocked → in-progress → done
in-progress → approved (deprioritized, back to backlog)
```

### Sprint Status
```
planned → active → completed
planned → cancelled
active → cancelled
```

## View Generation Rules

When any skill needs to **display** an aggregate view (backlog table, sprint board, status dashboard):

1. Read the index file (BACKLOG.md, SPRINT.md) for structure (ordering, wave grouping)
2. Read each referenced source file (feature/task) for live data
3. Display the merged result
4. **Never write source data back to the index file**

When any skill needs to **update** data:
1. Write to the source file only (feature or task frontmatter)
2. If the index file needs structural changes (reorder, add/remove ID), update the index
3. Do not propagate data fields to other files

## Deriving Epic Membership

To find all features in an epic:
```
Grep for `epic: E001` across $(shipyard-data)/spec/features/*.md frontmatter
```
Do NOT maintain a `features:` array in epic files. It's always computed.

## Deriving Feature Tasks

Feature frontmatter contains `tasks: [T001, T002, T003]` — the list of task IDs. Task details (title, effort, status) are read from individual task files. The feature file does NOT contain an inline task table.
